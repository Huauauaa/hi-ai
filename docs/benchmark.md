# 大模型的 Benchmark

**Benchmark** 是给大模型定的标准化考题：固定数据集、固定评测协议、固定指标。分数的意义是「在同一套规则下，模型 A 是否优于模型 B」——不是「模型有多聪明」的绝对判断。

榜单上的数字是压缩后的摘要，读之前先问：测的是什么能力、用什么 prompt、有没有数据污染、和你业务有多近。

## 常见能力维度

| 维度 | 代表 Benchmark | 主要测什么 |
| --- | --- | --- |
| **综合知识** | MMLU、CMMLU、C-Eval | 多学科选择题，覆盖面广 |
| **推理 / 数学** | GSM8K、MATH、AIME 子集 | 多步推导、算术与符号推理 |
| **代码** | HumanEval、MBPP、LiveCodeBench | 函数实现、通过率与执行结果 |
| **指令遵循** | IFEval、MT-Bench | 格式约束、多轮对话质量 |
| **长上下文** | Needle-in-a-Haystack、RULER | 长文检索与跨段信息整合 |
| **Agent / 工具** | SWE-bench、τ-bench | 真实环境里的任务完成度 |

没有「一个分数打天下」。MMLU 高不代表代码强，GSM8K 高也不代表产品对话好用。

## 重点指标解析

下面按「指标是什么 → 怎么算 → 怎么读」拆解常见榜上的核心数字。

### MMLU / CMMLU / C-Eval

| 项目 | 说明 |
| --- | --- |
| **指标** | Accuracy（准确率） |
| **题型** | 四选一选择题，覆盖 STEM、人文、社科等 50+ 学科 |
| **典型报告** | 总准确率；有时附分科（如 `mmlu_law`、`mmlu_medicine`） |

**怎么读：**

- 报告 **70%** 表示 100 道题里对了 70 道；不同 harness 的 **5-shot / 0-shot** 可差数个点到十几点。
- **CMMLU、C-Eval** 是中文/中国语境版，不能和 MMLU 分数直接比绝对值，只能同协议下横向比模型。
- 总分会掩盖偏科：法律高、医学低，对合规类产品比总分更有参考价值。

### GSM8K / MATH

| 项目 | 说明 |
| --- | --- |
| **指标** | Exact Match（答案精确匹配） |
| **GSM8K** | 小学难度应用题，~1.3k 测试题 |
| **MATH** | 竞赛级数学题，难度远高于 GSM8K |

**怎么读：**

- 模型输出常带推理链；评测一般只抽 **最终数字或表达式** 与标准答案比对，推理写错但蒙对答案仍算对。
- **CoT**（Chain-of-Thought）通常显著提升 GSM8K，对比榜单时必须看是否允许 CoT、是否 `maj@1`（多次采样投票）。
- GSM8K **90%+** 已是强模型常态；MATH 上 **50%** 已属顶尖，二者不可类比。

### HumanEval / MBPP（代码）

| 项目 | 说明 |
| --- | --- |
| **核心指标** | **pass@k** |
| **pass@1** | 每题生成 1 次，通过单元测试的比例 |
| **pass@k** | 每题生成 k 次，至少 1 次通过的概率（k>1 时更宽松） |

**怎么读：**

- 榜单默认多为 **pass@1**（更难、更常用）；论文里有时报 pass@10 / pass@100，数字会好看很多，对比前先看 k。
- 只测 **函数补全**，不含工程上下文、依赖管理和多文件修改；SWE-bench 才更接近真实开发。
- **MBPP** 题面更直白，HumanEval 偏算法；两者趋势一致但绝对值不同。

### HellaSwag / WinoGrande（常识）

| 项目 | 说明 |
| --- | --- |
| **指标** | Accuracy |
| **测什么** | 句子续写、指代消解等「常识下一句话」 |

**怎么读：**

- 对基座模型（pretrain）更有意义；Chat 模型在榜上的提升部分来自对齐，不代表对话产品体验。
- 题面偏英文日常场景，中文业务相关性弱，适合当 **基座质量** 参考，不宜当产品 KPI。

### BBH（BIG-Bench Hard）

| 项目 | 说明 |
| --- | --- |
| **指标** | 各子任务准确率的宏平均 |
| **测什么** | 从 BIG-Bench 挑出的 23 个对当时模型仍困难的推理任务 |

**怎么读：**

- 强调 **多步推理、逻辑、语言理解** 的「硬骨头」子集。
- 子任务差异大，建议看 **分任务** 而不只看宏平均；与 MMLU 知识题互补。

### IFEval（指令遵循）

| 项目 | 说明 |
| --- | --- |
| **指标** | **Strict** / **Loose** 遵循率 |
| **测什么** | 输出是否满足可自动校验的格式约束（字数、列表、关键词、语言等） |

**怎么读：**

- **Strict**：硬规则全满足才算对，更严。
- **Loose**：允许轻微格式偏差，分数通常更高。
- 对 **Agent、报告生成、结构化输出** 场景比纯知识题更有预测力；和「答对内容」正交——模型可以懂很多但格式不听话。

### Chatbot Arena（Elo）

| 项目 | 说明 |
| --- | --- |
| **指标** | **Elo 分**（相对排名分，无上限） |
| **怎么来** | 用户看两个匿名回复，投票谁更好；用对局结果更新 Elo |

**怎么读：**

- **1200 vs 1250** 的差距表示长期对局中的胜率优势，不是百分比。
- 反映 **人类主观偏好**（有用、安全、风格），和客观题库准确率不是一回事。
- 英文对话、公开用户分布；垂直领域（医疗、法律）排名可能失真。

### MT-Bench

| 项目 | 说明 |
| --- | --- |
| **指标** | 1–10 分均分（常由 GPT-4 等强模型当评委） |
| **维度** | 写作、角色扮演、推理、数学、编程、提取等 8 类 |

**怎么读：**

- **LLM-as-Judge**：评委模型变了，历史分数不可纵向比。
- 适合快速比较 Chat 模型 **多轮能力**；存在评委偏见（偏爱长回答、特定风格）。

### SWE-bench（真实代码修复）

| 项目 | 说明 |
| --- | --- |
| **指标** | **Resolved %** — 提交的 patch 让仓库测试由 fail 变 pass 的比例 |
| **测什么** | 根据 GitHub Issue 在真实代码库里修 bug |

**怎么读：**

- 远比 HumanEval 难，当前顶尖模型也只有 **个位数到二三十个百分点**。
- 区分 **Agent 框架 + 工具链** 与裸模型；报分时要说明是否带 browsing、是否限定步数。

### 指标对照速查

| 指标 | 常见范围 | 越高越好？ | 主要盲区 |
| --- | --- | --- | --- |
| MMLU acc | 60–90% | 是 | 选择题、英文中心、知识非应用 |
| GSM8K acc | 50–95% | 是 | 仅最终答案、数学非代码 |
| pass@1 | 30–90% | 是 | 单函数、无仓库上下文 |
| IFEval strict | 40–80% | 是 | 可规则化的指令 only |
| Arena Elo | 1000–1400+ | 是 | 主观、英文对话 |
| SWE resolved | 5–40% | 是 | 成本高、环境敏感 |

## 开发者应该关注的指标

论文和榜单服务研究对比；**日常开发**更应盯住「能不能发版、线上会不会炸、钱够不够烧」。下面按优先级分层。

### 1. 线上工程指标（必看）

这些不进学术榜，但决定用户体验和账单。

| 指标 | 含义 | 典型告警线（因业务而异） |
| --- | --- | --- |
| **TTFT** | Time To First Token，首 token 延迟 | 交互场景 p95 > 2s 需优化 |
| **E2E 延迟** | 整段回复完成时间 | 随输出长度线性涨，看 p95 |
| **吞吐** | tokens/s（生成阶段） | 批量任务看 GPU 利用率 |
| **成本** | 单次请求 / 每 1M token 费用 | 对比模型升级前后账单 |
| **错误率** | 5xx、超时、限流 | 应分模型、分路由统计 |
| **可用率** | SLA 内成功响应比例 | 和 fallback 策略联动 |

> 同样 MMLU 高 2 分，若 TTFT 翻倍或成本涨 3 倍，产品层未必接受。

### 2. 任务质量指标（按产品形态选）

公开 benchmark 是代理指标；**开发者应把它们映射到自己的任务定义上**。

| 你在做什么 | 优先关注的榜 / 指标 | 更应自建的指标 |
| --- | --- | --- |
| **对话 / 助手** | IFEval（格式）、MT-Bench 子项 | 多轮满意度、拒答率、幻觉抽检 |
| **代码补全 / Copilot** | HumanEval pass@1、LiveCodeBench | 仓库内采纳率、diff 接受率 |
| **代码 Agent** | SWE-bench resolved | Issue 闭环率、CI 通过率、回滚率 |
| **RAG / 知识库** | 领域 QA 集（如 CMMLU 相关科） | 检索命中率、引用准确率、无答案率 |
| **结构化输出** | IFEval strict | JSON Schema 校验通过率 |
| **Tool / Function Call** | τ-bench 类 Agent 榜 | 工具调用成功率、参数合法率、幂等重试率 |

核心原则：**公开榜做冒烟回归，业务指标做发版门禁**。

### 3. 迭代回归指标（每次改 prompt / 模型 / 路由）

写进 CI 或发版 checklist，变化超过阈值就拦：

| 指标 | 用法 |
| --- | --- |
| **冒烟子集准确率** | 50–200 条 held-out，对齐 [Harness](/harness) 固定协议 |
| **相对基线 Δ** | 新模型相对上一版的差值，而非绝对分 |
| **分桶表现** | 按语言、长度、难度拆开，防止总分掩盖退化 |
| **坏例数** | 从 85%→87% 不如「P0 场景 0 回归」 |
| **Judge 一致性** | 若用 LLM 打分，固定评委版本并抽人工校准 |

```text
发版条件示例：
  gsm8k_smoke     ≥ baseline - 1.0pt
  ifeval_strict   ≥ baseline - 0.5pt
  domain_heldout  ≥ baseline - 0pt   # 业务集不允许降
  p95_latency     ≤ baseline + 10%
```

### 4. Agent / 流水线专属

和 [算子](/operator) 视角对齐：每个 tool 都是可观测节点。

| 指标 | 说明 |
| --- | --- |
| **工具调用成功率** | 参数合法 + 执行无未捕获异常 |
| **计划步数 / 重试次数** | 反映模型是否「绕圈」 |
| **单次任务 token 消耗** | Agent 多轮放大成本 |
| **人工接管率** | 最终需人介入的比例，线上金标准 |

### 5. 可以少看的指标

| 指标 | 原因 |
| --- | --- |
| **单一 MMLU 总分** | 与多数产品任务弱相关，易刷 |
| **HellaSwag**（对齐后 Chat 模型） | 对应用层决策帮助有限 |
| **跨期 Arena Elo 绝对值** | 用户分布变、模型匿名池变，纵向难比 |
| **pass@100**（未注明 k 时） | 与 pass@1 不可比，易误导 |
| **他人私有测试集** | 无法复现、无法回归 |

### 6. 推荐最小仪表盘

**每次模型 / prompt 变更都看：**

- [ ] 3 个公开冒烟任务（与业务最接近的）
- [ ] 1 套领域 held-out（≥50 条）
- [ ] p50 / p95 延迟 + 单次成本
- [ ] 错误率 & 超时率
- [ ] 结构化 / 工具场景的规则校验通过率

**每月或重大版本再看：**

- [ ] 人工抽检 100 条（分场景分层）
- [ ] 线上坏例聚类（哪类 query 在变差）
- [ ] 成本 per successful task（成功任务单位成本）

## 读分时要看的细节

1. **Prompt 协议** — 0-shot 还是 5-shot？CoT 是否允许？换 prompt 排名可以洗牌。
2. **模型版本** — API 模型是否注明 snapshot（如 `gpt-4-0613`）？未 pin 版本的结果不可复现。
3. **评测实现** — 是否用 [Harness](/harness) 或公开 harness 跑？自写脚本的结果难横向对比。
4. **数据污染** — 训练集是否见过 benchmark 原题？公开集高分要警惕 memorization。
5. **置信区间** — 小样本集上 1–2 分的差距可能只是噪声。

## 主流榜单与工具

- [Open LLM Leaderboard](https://huggingface.co/spaces/open-llm-leaderboard/open_llm_leaderboard) — 社区模型综合榜，任务组合会变，需看当期说明。
- [LMSYS Chatbot Arena](https://chat.lmsys.org/) — 人类偏好 Elo，更贴近对话体验，成本高、主观性强。
- [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) — 本地复现学术榜的常用入口（见 [Harness](/harness) 一文）。

```bash
# 示例：同时跑多个任务
lm_eval --model hf \
  --model_args pretrained=Qwen/Qwen2.5-7B-Instruct \
  --tasks mmlu,gsm8k,hellaswag \
  --batch_size auto
```

## Benchmark 与业务的关系

公开 benchmark 是**筛选器**，不是**验收单**：

- **筛选** — 缩小候选模型范围，排除明显短板。
- **验收** — 必须用业务自己的 held-out 集、线上 A/B、人工抽检。

建议维护三层评测：

| 层级 | 内容 | 频率 |
| --- | --- | --- |
| **公开榜子集** | MMLU / GSM8K 等冒烟任务 | 每次模型或权重变更 |
| **领域集** | 内部 query、工单、代码库片段 | 每周或每次发版 |
| **线上指标** | 延迟、成本、成功率、用户反馈 | 持续 |

## 常见误区

- **刷榜即产品** — 针对公开集过拟合 prompt 或微调，线上未必增益。
- **只看总分** — 子任务（如 MMLU 的法律/医学）才暴露真实短板。
- **忽略成本** — 70B 比 7B 高 5 分，但推理贵 10 倍，业务可能不划算。
- **混用 API 与开源** — 温度、系统提示、安全过滤不一致时，数字不可比。

## 最小实践

- [ ] 选定 3–5 个与业务最相关的公开任务，写入 CI 冒烟
- [ ] 记录模型 revision、温度、max_tokens、prompt 模板版本
- [ ] 自建 50–200 条领域题作为 held-out，不进入训练
- [ ] 发版前对比上一版的**分任务**差异，不只看平均分
- [ ] 榜单分数与 [Harness](/harness)、[算子](/operator) 流水线一起版本化
